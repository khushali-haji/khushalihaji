The Problem

Understanding a conflict shouldn't take 12 browser tabs

When an international event breaks, a single news report gives you fragments, not understanding. The location is buried in one paragraph, the actors in another. The historical backdrop is assumed. And the article's own media slant is invisible unless you already know the outlet's political positioning.

So an engaged reader does the synthesis by hand: open a second map tab to locate where the action occurred, skim Wikipedia to understand the history, and load a media-bias tool to detect the narrative framing. Three tools, ten minutes, and most readers give up before starting. Worse, the most rigorous reporting sits behind paywalls.

How might we take someone from a single article to a geographically-grounded, bias-aware understanding of an event in under a minute?

Research & Insight

I started with the workflow, not the interface

Who it's for

The core user isn't a headline-skimmer — it's the engaged reader: students, analysts, policy makers, and globally-curious individuals. Their bottleneck is not a lack of information; it's a lack of synthesis. The data is available, but the cognitive labor of assembling it is exhausting.

From insights to principles

The user workflows guided four core product pillars:

01

Geography is the spine

The map leads, the text supports. Coordinates are the anchors that hold narrative fragments together.

02

Never make them wait

Stream core details in 5s. Let deep LLM analysis load progressively in the background.

03

Make bias visible

Source framing is a feature to read skeptically, not an invisible bias to fall victim to.

04

Meet readers where they are

Provide secondary alternative uploads (like PDFs) to bypass scraper paywall obstacles.

Defining the Product

One flow, three moves

To maintain product magic, I kept the scope extremely narrow. No user accounts, no saved dashboards, and no feed pages. The entire product is optimized for a single 3-step sequence: **Input link or PDF → Run processing → Explore interactive briefing.**

The Desktop Interface

Three screens, one continuous narrative

Screen 1 — Input

A quiet, editorial entry point: warm neutrals, soft sage highlights, and a clean serif title layout. A simple two-tab selector allows pasting an article URL or dragging in a raw PDF file.

Screen 2 — Analysis

Rather than a generic spinner, the wait screen uses precise copy indicating what stage is processing: "Extracting primary incidents and locating actors (~5-10s)..." Honest state indicators manage expectations and lower perceived wait times.

Screen 3 — The Briefing Dashboard

A dark, high-contrast satellite map fills the background, with clean glassmorphic panels floating on top. Map boundaries automatically adjust on load to display all georeferenced events, with timeline markers matching the color-coded map pins.

Mobile Interaction Model

The desktop pattern breaks on a phone — so I changed the metaphor

Floating overlays on desktop maps are elegant, but they cover the entire viewport on a mobile screen. Instead of wrapping overlays in tiny scroll zones, I refactored the layout into a **Map-App Pattern**. The map remains the base layer, and a bottom sheet slides up dynamically.

Key Design Decisions

Decisions, and the tradeoffs behind them

Progressive rendering over single payload

Tension: Single API payloads are cleaner to maintain, but force a 25-second blank screen.

Decision: Segment the LLM pipeline into four distinct threads, painting the UI sequentially.

↳ Core Principle: Never make them wait

A PDF bypass for scraper paywalls

Tension: Scrapers cannot read premium publications without encountering subscriber login screens.

Decision: Built a PDF upload parser so users can save paywalled pages and process raw text directly.

↳ Core Principle: Meet the reader where the content is

Standalone bias diagnostic panel

Decision: Introduced a dedicated "Journalistic Analysis" panel showing institutional lean and regional slants.

↳ Core Principle: Make bias visible

Designing for Unreliable AI

The hardest design problem wasn't the UI — it was the AI

Language models are probabilistic systems. They can return malformed JSON, hallucinate URLs, or geocode coordinates to the middle of the Atlantic Ocean. Designing for AI requires building defensive UI safety nets around messy data.

Geocoding Coordinates

"lat": 0.0, "lon": 0.0 (Lands in the ocean)

Hallucinated Outbound Links

The model consistently invented article URLs that 404'd. Rather than listing dead links, the backend prompts the model for the outlet name and incident search query, constructing verified Google News search links instead.

Ocean Coordinates

Instead of placing coordinates at 0.0, 0.0, the pipeline filters invalid coordinate bounds and centers pins at the capital city of the reported country, flagging coordinate fallback notices in the UI.

Good UX for AI is as much about handling failure as showing off success.

Technical Architecture

Under the hood

Frontend Framework
React + Vite, React-Leaflet map overlays (greyscale satellite tile imagery), and customized CSS sheets.
Backend Engine
Node + Express hosting, orchestrating parallelized streams to Google Gemini for content parsing.
Scraper Pipeline
Cheerio & Axios text extraction, pdf-parse modules, and a temporary Redis cache to bypass redundant scraping.
Deployment
Frontend statically served via GitHub Pages, with APIs operating on standard cloud server runtimes.

Reflection

What I learned, and where it goes next

Conflict Lens taught me that designing with AI is fundamentally about designing for volatility. We cannot control the API outputs, so the UI must act as a filter. Managing system speed through progressive skeleton loading, setting realistic expectations with clear waiting copy, and offering failover actions are what transform a fragile AI script into a robust product people trust.

If I were to expand the project scope: I'd build a chronological playback tool that animates the timeline sequence, flying the camera dynamically to each event pin on the map. I would also add keyboard shortcuts to browse events, and implement institutional trust scores to verify citations.
